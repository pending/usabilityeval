Help users Recognize, Diagnose, and Recover from Errors

Error messages should be expressed in plain language (no codes), precisely indicate the problem, and constructively suggest a solution.

The error page users visit after a stumbling upon a bug in the system provides ample user personality -- enough interactive dialogue to maintain a friendly and apologetic point of view. The website provides a sufficient bright red "Not Found" message to let the user know that they are, indeed, on the wrong page. However, two of the links they provide to help the user re-direct to somewhere useful do not even work. This is a terrible way of errory recovery, as it just leads to another error. On the plus side, the A-Z listing works!

Key Example:

http://www.le.ac.uk/hello

For their other website (when it is www2), it provides more useful information, such as related articles. This page proves more useful than the other error page, because the links actually work, and they provide relevant information based on the faulty URL the user entered.

Key Example:

http://www2.le.ac.uk/dog
